# CORS
- 악의를 가진 해커로부터 인터넷을 보호하기 위해 만들어 짐
- 다른 출처의 리소스 공유에 대한 허용/비 허용 정책
- SOP 정책을 위반해도 CORS 정책에 따르면 다른 출처의 리소스라도 허용한다는 의미
- SOP
    - 동일한 출처에서만 리소스를 공유할 수 있음
    - 다른 출처의 서버에 있는 이미지나 유튜브 영상 같은 리소스는 상호작용이 불가능함
- Origin
    - HOST + 
## 브라우저의 CORS의 기본동작
- 클라이언트에서 자신이 보냈던 요청의
- CORS 해결책은 서버의 허용이 필요
- 서버에서 Access-Control-Allow-Origin 헤더에 허용할 출처를 기재해서 클라이언트에 응답
- CORS 에러가 발생하면 브라우저 개발자 도구(F12) -> 네트워크 탭 -> 요청/응답 헤더를 확인하여 어떤 원인인지 파악하는 것이 중요

## CORS 에러가 발생하는 경우
- 출처가 다른 도메인 또는 포트로 리소스를 요청할 때
- HTTPS에서 HTTP로 리소스를 요청할 때
- 인증 정보를 포함한 요청을 보낼 때
- 허용되지 않는 도메인 사용할 때
- 허용되지 않는 헤더 사용할 때
- 프리플라이트 요청에 대한 응답이 없는 경우
- 브라우저가 CORS를 지원하지 않는 경우
- CDN 등에서 제공하는 리소스를 요청할 때

## CORS 작동 방식 3가지
- 예비 요청
    - Options 도 처리해야됨
- 단순 요청
    - 아래 3가지 필수 충족해야 됨
    - 요청의 메소드는 GET, HEAD, POST 중 하나여야 함
    - 특정 헤더일 경우에만 적용
    - Content-Type 헤더가 특정 헤더 중 하나여야 함
- 인증된 요청
    - 인증 관련 헤더를 포함할 때 사용하는 요청
    - JWT 인증 토큰을 주고받을 때 등에 사용

## CORS 보안문제 예방 가이드
- 와일드카드(*) 출처 사용 금지
    - Access-Control-allow-origin: * (X)
    - Origin 요청 헤더의 값을 그대로 사용 금지
    - NULL 출처 허용 금지
    - 화이트 리스트 사용
